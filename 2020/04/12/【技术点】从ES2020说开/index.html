【技术点】从ES2020说开去

ES是什么

ECMAScript

ES指的是ECMAScript标准。ES标准全文可见

一开始叫做JavaScript，后来为了推动JavaScript的发展，网景公司将JavaScript提交给欧洲计算机制造商协会(ECMA)进行标准化。

ECMA存在的意义就在于统一不同JS引擎执行的结果。

除了「ECMA」这个制造商协会的概念之外，还有一个「TC39」的掌权协会。

TC39

TC39是ECMA的第39号技术专家委员会（Technical Committee 39）的简称，是ECMAScript标准的实际制定者。

这个委员会并不是一个人，而是负责制定ECMAScript标准的一群人，其成员主要由各个JS引擎（浏览器）厂商工作人员组成。

事实上，任何人或开发者都可以注册成为一个TC39贡献者，并通过其Github提交proposal或编写测试用例。

提交网址如下：https://github.com/tc39/proposals/blob/master/stage-0-proposals.md

提案

一个提案要历经四次磨难才能最终成为标准。

  • Stage 0（Strawman阶段）
    • 该阶段是一个开放提交阶段，任何在TC39注册过的贡献都或TC39成员都可以进行提交。提交内容可以是一个提议、想法、初步描述。
  • Stage 1（Proposal阶段）
    • 该阶段是对所提交新特性的正式建议。在这个阶段需具备以下条件：
      • TC39指定一名成员作为Champion
      • TC39审阅通过
      • 有实现的 Demo 或者 Polyfill
    • 初步编写标准，包括：问题描述、解决方案示例、语法语义API、关键的算法及抽象、实现的复杂度等
  • Stage 2（Draft阶段）
    • 该阶段是会出现标准中的第一个版本。在这个阶段必须要具备以下条件：
      • 更规范化语法特性和语意的说明，并使用ECMAScript标准的正式语言描述该特性的语法、语义、API
      • 该特性应该有两个实验性的实现，其中一个可以基于Babel等转译器实现
  • Stage 3（Canidate阶段）
    • 该阶段的提议已接近完成，只需要得到提议实现方的反馈，并由用户来进一步推动。在这个阶段应具备以下条件：
      • 编写Test 262测试例，且至少2个用例通过测试
      • TC39指定的审稿人审核通过
      • 开发者的足够认可
  • Stage 4（Finished阶段）
    • 该阶段的会被包括到标准之中。在这个阶段应具备以下条件：
      • Test 262验收测试通过
      • 两个符合规范的实现通过
      • TC39审核通过
      • 开发者的支持和认可

接下来，经过ECMA的审核后，该提议将正式被添加到标准中。一般来说只有到了Stage 4，该特性才会被确定加入到标准中，然后会在下一次的ECMAScript标准中出现该提议， 当然也并非是百分百的，也可能需要更长的时间。

所以我们使用了很多方法在一段时间后才正式纳入了标准

ES发展史

ES标准早在1996年就出现了第一版，然后保持一年一版的更新速度更新到ES3，后来停滞了一段时间，提出了一个激进版的更新ES4，被一部分人反对就不了了之了。其中有些被接受的小修改被命名为ES3.1后来改名叫ES5。ES4里的一些步子比较大的更新就放到了ES6里，ES6之后又恢复了一年一版本的节奏。

ES6是一个分界线，因为一些利益的冲突，话语权公司的争执不下，ES4流产只进行了小范围的改良, 从ES3升级到ES5，ES4的一些激进的优化都放到了ES6及以后的版本。
ES6的更新主要包括：

  • 类
  • 模块化
  • 箭头函数
  • 函数参数默认值
  • 模板字符串
  • 解构赋值
  • 延展操作符
  • 对象属性简写
  • Promise
  • Let与Const

像promise、let、箭头函数、解构等已经变成了现在使用最普遍的特性了。

ES2020的新特性

ECMAScript 2020, the 11th edition, introduces the matchAll method for Strings, to produce an iterator for all match objects generated by a global regular expression; import(), a syntax to asynchronously import Modules with a dynamic specifier; BigInt, a new number primitive for working with arbitrary precision integers; Promise.allSettled, a new Promise combinator that does not short-circuit; globalThis, a universal way to access the global this value; dedicated export * as ns from ‘module’ syntax for use within modules; increased standardization of for-in enumeration order; import.meta, a host-populated object available in Modules that may contain contextual information about the Module; as well as adding two new syntax features to improve working with “nullish” values (null or undefined): nullish coalescing, a value selection operator; and optional chaining, a property access and function invocation operator that short-circuits if the value to access/invoke is nullish.

BigInt

由于在JS中数据都是以双精度浮点数的形式存在.因此小数计算可能出现精度的误差，大数的计算和表达中有安全数是2^53-1的概念。

为了避免这种情况，可以使用math.js等第三方库进行运算，但是他们大多是将数值看作是字符串从而进行溢出风险的规避。字符串的运算操作，性能自然没有原生数据好。

在2018年就已经有了BigInt的提案，chrome在67版本的时候就提供了支持。但是今年才正式进入了标准。

在标准中，BigInt和Number一样都是基本类型。

BigInt可以与Number/String进行显式转换但不会被隐式转换，因此像于BigInt与Int的等于/小于之类的操作并没有隐式转换的过程。

由Number(BigInt)进行转换还是会造成精度的损失；String(BigInt)不含n。

BigInt只是个函数，没有构造器，因此不能使用new来创建 BigInt的实例

Nullish

回顾之前，当我们使用 ||，其实我们想做的是为变量赋一个初始值

但是在Boolean的判断中除了null和undefined之外「0」、「‘’」、「false」也会被看作是false,就会出现当变量的值是「‘’」也会被直接忽略而取后面设置的默认值。
因此，在ES2020将「??」纳入了标准。只有当变量的值是null/undefined才看作是「假」。

Optional chain

在前后端交互过程中，当想要获取对象深层的属性的时候要首先判断对象（同样适用于数组和函数）是否存在

当对象的深度比较浅的时候可以通过每一层级的对象存在与否进行判断，如果层级较为复杂就会增加代码阅读的负担。因此，ES2020将Optional chain纳入了标准，通过「?.」的操作符号，先进行变量存在与否的判断再进行属性/方法的取值。

当需要指定默认值时，Optinal Chaining 操作符还可以与nullish操作符（「??」）组合使用。

Promise.allSettled

当页面与多个请求结果有关时，为了避免因为一个reject而导致所有的请求都失败，新标准里纳入了「Promise.allSettled」。Promise.allSettled返回一个promise，包含promise的结果。

此外，现在处于stage-3阶段的还有一个promise方法，promise.any。promise.any不管是成功还是失败都会得到一个最先执行完的promise，存在first里，但只有当所有的都失败了才会被当作reject. 对比promise.race,第一个执行完成后续的不管了.

For-in

新标准中规范了for-in的遍历顺序。首先将所有属性进行praseInt的操作，然后属性为整数的key按照key的大小进行排列，剩下的属性按照创建顺序遍历。

纳入标准的原因应该是之前不同的浏览器在for-in上的遍历顺序不一样，但是还没对这个猜测进行验证

此外，需要注意的是使用for in会遍历数组所有的可枚举属性，除了Symbol类型之外，包括原型链。
如果只想遍历对象本身的属性可以用hasOwnproperty进行过滤。

常见的遍历的方式有

for in (得到key，还要处理原型链)

for of (得到value，对象不可用，针对数组之类的iterable元素)

Object.keys (得到自身的所有属性,需要注意的是Object.keys得到的key的顺序和for in的顺序相同)

Object.getOwnPropertyNames （含不可枚举的属性）

Object.entries （和for-in一样的顺序一样，得到的是含key和value的元组）

所以如果想要按照创建顺序调用避免使用可以Number或是可以转化为Number的key值

matchAll

matchAll是为了返回字符串中所有匹配的信息，返回的是一个迭代器。对比之前的string.match和regExp.exec()方法

在一些博客里看到这样的说法，「如果没有/g标志，matchAll返回第一个匹配」，通过验证发现并不是。

string = "test ya test"
result = string.matchAll(/test/) 

=> VM1955:2 Uncaught TypeError: undefineds called with a non-global RegExp argument
    at String.matchAll (<anonymous>)
    at <anonymous>:2:17

查看标准有这么一个说法：
If ? ToString(flags) does not contain “g”, throw a TypeError exception.
因此可以看出，只要不含g，就会抛出异常，而regExp.exec()里的正则如果不含「/g」则会一直在第一个匹配上循环。

总结
  • match(/xxx/) 返回一个数组，含有匹配值、index、input、groups
  • match(/xxx/g) 返回一个数组，仅有匹配值
  • matchAll(/xxx/g) 返回一个迭代器，迭代器里的value与match的内容相同，done判断是否匹配完成
  • regExp.exec(str); // 返回一个数组与match相同，但是会记录index的位置，再次执行就会按序匹配

import()

本质是返回一个promise。

按需 import 提案几年前就已提出，如今终于能进入ES正式规范。现代前端打包资源越来越大，打包成几M的JS资源已成常态，但是前端应用初始化时往往不需要全量加载逻辑资源，为了首屏渲染速度更快，很多时候都是按需加载，比如懒加载图片等。
因为import命令被JS引擎静态分析，先于模块内的其他语句执行，无法取代require()的动态加载功能，提案建议引入import()来代替require()

可以参考 翻译 | 原生 ECMAScript 模块：动态 import（）

globalThis

每个环境都有其自己的对象模型，并提供了不同的语法来访问全局对象。 例如，在 Web 浏览器中，可以通过window，self或frames(此只读属性通常用于获取当前窗口的子帧列表。例如，我们可以使用window.frames[0]或frames[0]来访问第一帧。)访问全局对象。 但是，在 Node.js 中，这些属性不存在，而必须使用global。 在Web Worker中，只有self可用。

  • Browser：顶层对象是window
  • Node：顶层对象是global
  • WebWorker：顶层对象是self
    这些不同的全局对象引用方法让 JS 实现跨平台变得非常困难。因此globalThis出现了。
