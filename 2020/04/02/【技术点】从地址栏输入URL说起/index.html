【技术点】从地址栏输入URL说起

Posted on 2020-04-02 | Post modified: 2020-04-04
Words count in article: | Reading time ≈

概括来说

主要有以下5个部分：

  1. DNS解析
  2. TCP连接
  3. HTTP请求
  4. 服务端响应
  5. 浏览器解析请求资源

每个部分又可以展开说去，以上也是影响首屏加载速度的因素。

DNS解析

目的是通过输入的url中的域名获取对应的IP地址，因为IP地址比较复杂且没有一定的规律性，所以使用中多是通过域名进行IP地址的映射。DNS使用TCP和UDP端口53。

已知域名查找IP地址的过程叫做DNS解析，负责解析工作的服务器叫做DNS服务器。

DNS服务器

  • 根域名服务器是互联网域名解析系统（DNS）中最高级别的域名服务器，负责返回顶级域的权威域名服务器地址。
  • 顶级域名服务器
  • 次级域名服务器
  • 主机名服务器

递归还是迭代

  • 递归的意思是当找不到这个域名的时候依次按级转发到上层DNS服务器，上级又消息再返回含有该映射关系的域名服务器给请求者。（自下而上）
  • 迭代的意思是当找不到这个域名时直接发到根DNS服务器，由负责该域名的顶级服务器进行请求返回含有该映射关系的域名服务器。（自上而下）

请求过程

  1. 首先在浏览器中查找有没有对应的缓存
    1. 浏览器在获取网站域名的实际IP地址后会对其IP进行缓存，减少网络请求的损耗。
    2. 每种浏览器都有一个固定的DNS缓存时间
  2. 如果没有命中，则去查找本机的hosts文件
  3. 如果没有命中，则去本地的DNS服务器上去寻找是否有相应的缓存
    1. 域名缓存的最长时间由请求中的TTL参数决定
  4. 如果没有对应的缓存，本地DNS服务器就去本地配置区域资源中筛选
    1. 本地DNS服务器为TCP/IP参数中设置的首选DNS服务器（权威DNS服务器）
  5. 如果还是没有找到对应的结果，就通过本地DNS的设置去决定是递归查找还是迭代查找。
  6. 直到返回对应映射关系

TCP连接

获取了对应域名的IP就根据URL中含有的端口号，建立TCP连接。

TCP建立过程

三次握手
  1. client->Hello(SYN=1,seq=i)
  2. server->ACK(ack=i+1,SYN=1,seq=j)
  3. client->ACk(ack=j+1)

为什么偏偏是三次握手


两次握手不可靠，client可能没有收到server的ack回复导致server处连接被建立但是一直没有进行数据的交互。


四次握手会浪费资源，server和client端都通过ACK进行了状态的确定就没有必要再进行一次ACK的确定了。

四次挥手
  1. client-> FIN(C:FIN-wait1|S:close_wait)
  2. server-> ACK(C:FIN-wait2|S:close_wait)
  3. …(未传递完的数据的传输)
  4. server-> FIN(C:Time_waiting|S:last_ack)
  5. client-> ACK(C:null|S:Closed)(server没有接收到此ACK的话会在N秒后重发，以避免FIN/ACK丢包的情况)

为什么偏偏是四次挥手


四次挥手是绅士的挥手，也可以2次挥手，当client接收到server的ACK就中断连接，但是这样会导致server中断其还没传输完的数据。

TCP与UDP
  1. TCP是面向连接的，UDP是面向报文的
  2. TCP有拥塞控制有重传机制是可靠的；UDP不可靠、不会进行差错控制
  3. 但是UDP实用性好，工作效率高

HTTP请求

TCP属于传输层协议，HTTP属于应用层协议。

HTTP传输报文时，以流的形式将内容按序传输，TCP收到数据流之后，会将数据流分割成符合条件的小的数据块，并将段封装在 IP 分组中，通过数据链路层进行传输。

IP头[TCP头[HTTP头]]

所以这也就出现了,http2中头部压缩的方式，因为每一个IP包里都有重复的HTTP头，占据了传输的资源。

HTTP连接最显著的特点是客户端发送的每次请求都需要服务器回送响应，在请求结束后，会主动释放连接。从建立连接到关闭连接的过程称为“一次连接”。（短链接）

在HTTP1.x之后可以通过配置keep-alive实现长连接。

HTTP报文

请求报文
  1. 请求行
    1. 请求方法（GET/POST/DELETE/PUT/HEAD/OPTION）
      1. GET与POST的差别
        1. 数据是否在URL上可见[存储在浏览器缓存]、
        2. 由于浏览器要求导致GET的长度有限制、
        3. 是否幂等、
        4. 在之前chrome<80的时候SameSite的默认值为Lax，仅get的时候携带cookie、
        5. 需要注意的是http都是明文GET、POST都可见
      2. OPTION的作用（ 发送预请求 / 如果是简单请求服务端会根据请求头的origin字段判断是否同意 ）
    2. 请求URI
    3. 协议名及版本号
  2. 请求头
    1. Accept-Encoding
    2. Accept-Language
    3. Cache-control(no-cache / max-age)
    4. Connection（keep-alive / close）（长连接）
    5. Cookie
    6. Content-length(报文中主体部分的长度)
    7. Referer
  3. 空行
  4. 请求主体
    响应报文
  5. 状态行
    1. 协议名及版本号
    2. 状态码
      1. 1XX信息型
      2. 2XX成功型（200 / 204）
      3. 3XX重定向（301 / 302[临时重定向，POST改GET] / 307[不改变method] / 304）
      4. 4XX客户端错误（400 / 401[没带认证信息或者错误] / 403[认证的用户没有访问权限] / 404）
      5. 5XX服务器错误（500 / 502 / 503 / 504）
  6. 响应头
    1. Content-Type
    2. Connection
    3. Content-Encoding
    4. etag（与浏览器协商缓存有关）
    5. last-modified
    6. access-control-allow-origin（以下均与跨域有关）
    7. access-control-allow-Credentials
    8. access-control-allow-methods
    9. access-control-allow-headers
  7. 空行
  8. 响应报文主体

HTTP与HTTPS

HTTPS是在TCP与HTTP之间添加了一层SSL，起到了加密的作用。

  1. client-> Hello (随机数A，支持的协议、加密算法等)
  2. server-> Hello (随机数B，确定的协议和算法，服务端的证书)
  3. client-> Auth (利用公钥解密证书判断真伪，公钥加密随机数C和之前信息的hash，将A+B+C利用算法合并为对称加密的密钥切换到加密套件的状态)
  4. server-> Auth （用计算得到的对称密钥加密之前的hash）

    HTTP演进史

  5. 第一阶段 HTTP/1
    1. 由于连接无法复用，每一次连接都要进行三次握手和四次挥手，对大量小文件的请求影响较大。虽然支持keep-alive但1.1版本才变成官方标配
    2. 浏览器最大请求数量会导致队头阻塞，当浏览器需要请求很多资源的时候，资源队列里用时只有X个资源可以同时请求，剩余请求就必须要等待
    3. header中的重复内容过多增大了协议的开销
    4. 协议信息都是明文的
  6. 第二阶段 SPDY 协议
    1. 谷歌开发SPDY并不是为了代替Http1而是对http进行了增强，ssl上http下，强制使用ssl
    2. SPDY引入流和帧的概念，允许同一域名的请求在一个连接上进行并发传输，这样就减少多次TCP连接带来的损失；每个帧都是二进制有利于数据的传输和压缩。
    3. 但是请求的优先级有高有低，共用同一个连接可能会导致请求阻塞，因此引入了请求优先级概念，避免高优先级的请求被挂起
    4. 压缩请求报文头和响应报文头
    5. 支持服务器推送，通过X-Associated-Content头告诉客户端服务器将在客户端请求资源之前，推送资源给它。
  7. 第三阶段 HTTP/2
    1. SPDY在chrome上实验的效果很好，因此被当作了HTTP/2的基础
    2. 其实HTTP/2对于http协议是有兼容的，但是想chrome、firefox等都支持https版的HTTP/2
    3. 优化了报文头压缩算法，SPDY使用的是赫夫曼编码(即根据出现频率进行字母的替换)和LZ77编码(即根据相同内容的位置和大小进行压缩)；HTTP/2使用的是HPACK算法（静态字典+动态字典+赫夫曼编码[都匹配不上的Huffman编码一下]）
  8. 第四阶段 HTTP/3（QUIC）
    1. 考虑到丢包重传的问题，由于TCP的重传机制会导致丢失包后的包拥塞影响效率，HTTP/2在丢包重传上的表现甚至不如HTTP/1，因此Google提出了基于UDP的QUIC协议
    2. 前向纠错，丢包可以单独发送/不重传
    3. 利用ID标识每个连接

      附：websocket

      websocket是H5的一下实现API，支持持久的连接，双向通信。
      此过程从客户端向服务器发送常规 HTTP 请求开始。 此请求中包含 Upgrade 标头，通知服务器客户端希望建立 WebSocket 连接。

服务端响应

服务器接收到浏览器的请求后进行资源的响应。

浏览器解析请求资源

浏览器接收资源到将资源展示到页面上主要经过以下几个步骤。

  1. 加载HTML文件
  2. 自上而下解析HTML文件构建DOM树
    1. JS单线程,尽可能让内容和样式先显示出来，
    2. script(defer/async/js文件放在body后),避免阻塞DOM树的构建
  3. 解析CSS构建CSSOM树
  4. DOM和CSSOM结合生成render树
  5. 布局render树
    1. 重排/回流，都是由于render树的发生了更改
    2. DOM改变/元素尺寸改变/文本改变/窗口尺寸改变/激活伪类
    3. CSS改变会重新生成render树，因此对于样式频繁修改的元素可以利用浮动布局，使其脱离文档流 / 利用CSS transfrom使用硬件加速
    4. 重排一定会引起重绘
  6. 绘制
    1. 当背景颜色/字体颜色发生改变时会触发重绘
